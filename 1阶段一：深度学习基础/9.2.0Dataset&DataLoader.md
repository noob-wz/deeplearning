# PyTorch 数据管道：`Dataset` 与 `DataLoader`

## 1. 为什么需要数据管道

在玩具代码里，我们常常直接写：

```python
x = torch.tensor([[1.0], [2.0], [3.0]])
```

这种写法只是为了演示网络结构和张量计算，本质上是“凭空造数据”。

但在真实工程里，数据绝不会这样出现。

以计算机视觉任务为例，数据通常是：

* 几十万到上百万张图片
* 存在硬盘而不是内存里
* 总体积可能达到 GB 甚至 TB 级别

这会带来两个根本问题：

### 问题 1：一次性加载全部数据，根本存不下

如果试图把所有图片一次性读入内存：

```python
x = load_all_images()
```

很容易直接导致：

* RAM 爆掉
* 显存爆掉
* 程序崩溃（Out of Memory）

### 问题 2：GPU 算得快，硬盘读得慢

GPU 做矩阵运算很快，但硬盘读取图片、CPU 解码图片、预处理图片很慢。
如果训练时你手动写一个循环，一张一张去读图，再拼 batch，那么 GPU 算完当前数据后就只能空等下一批数据准备好。

于是会出现：

* GPU 利用率很低
* 训练速度很慢
* 计算资源被严重浪费

---

## 2. 设计动机：把“数据定义”和“数据搬运”分开

为了同时解决：

* **数据存不下**
* **数据搬得慢**

PyTorch 把数据处理拆成两个独立抽象：

### `Dataset`

它是**数据说明书**。

它不把所有数据都读进来，只负责回答两个问题：

1. 这个数据集一共有多少条数据？
2. 如果给我一个索引 `index`，我如何只取出这 1 条数据，并把它变成张量？

---

### `DataLoader`

它是**数据搬运引擎**。

它拿到 `Dataset` 之后，负责：

* 决定按什么顺序取样本
* 是否打乱顺序
* 一次取多少条组成一个 batch
* 是否用多个 worker 并行准备数据
* 如何把单条样本拼接成 batch

---

## 3. 一句话理解两者分工

### `Dataset`

**定义单个样本怎么取。**

### `DataLoader`

**定义很多样本怎么被组织成一批一批地送给模型。**

---

## 4. 为什么这样设计是合理的

这样设计的本质是：**按需加载，而不是一次性全加载。**

训练时不是：

* 先把所有数据全读进来
* 再开始训练

而是：

* 需要这一批时，才去读这一批
* 读完就交给 GPU 计算
* 同时后台继续准备下一批

于是形成流水线：

* **硬盘 / CPU**：负责读数据、预处理、拼 batch
* **GPU**：负责计算

两边尽量并行工作，不互相等待。

---

# 5. `Dataset` 是什么

## 本质

`Dataset` 是一个**按索引访问单样本的数据接口**。

在最常见的 map-style dataset 里，它通常需要实现两个方法：

```python
__len__(self)
__getitem__(self, index)
```

---

## `__len__`

返回数据集总长度。

作用：

* 告诉外界这个数据集有多少条
* 告诉训练循环一个 epoch 什么时候结束

例如：

```python
def __len__(self):
    return 1000
```

表示这个数据集有 1000 条样本。

---

## `__getitem__`

根据索引返回**单条样本**。

例如：

```python
def __getitem__(self, index):
    x = ...
    y = ...
    return x, y
```

意思是：

* 如果外界要第 `index` 条样本
* 你就读取这一条
* 返回它的输入 `x` 和标签 `y`

---

## `Dataset` 的核心思想

它**不负责 batch**，只负责：

> “给我一个索引，我返回一条样本。”

这点非常重要。

---

# 6. `DataLoader` 是什么

## 本质

`DataLoader` 是一个**批量数据迭代器**。

它接收一个 `Dataset`，然后在训练时持续产出：

```python
x_batch, y_batch
```

也就是说，它不是返回一条样本，而是返回**一批样本**。

---

## `DataLoader` 主要负责什么

### 1. batching

把多条样本拼成一个 batch。

### 2. shuffle

决定每个 epoch 是否打乱样本顺序。

### 3. multi-process loading

使用多个 worker 并行准备数据。
这里更准确地说，`num_workers > 0` 时通常是多个**子进程**，不是普通 Python 线程。

### 4. collate

把多个单样本整理成一个 batch。

---

# 7. `Dataset` 和 `DataLoader` 的协作流程

假设你定义了一个图像分类数据集：

* 单张图片 `x` 的形状是 `(3, 224, 224)`
* 单个标签 `y` 是类别编号

那么流程是这样的：

## 第一步：`Dataset`

如果调用：

```python
dataset[8]
```

它会返回第 8 条样本，例如：

```python
x.shape == (3, 224, 224)
y.shape == ()
```

或者：

```python
y.shape == (1,)
```

取决于你怎么写标签。

---

## 第二步：`DataLoader`

如果：

```python
dataloader = DataLoader(dataset, batch_size=32, shuffle=True)
```

那么 `DataLoader` 会：

1. 生成 32 个索引
2. 分别调用 32 次 `dataset.__getitem__(index)`
3. 拿到 32 个 `(x, y)`
4. 把它们沿着第 0 维堆叠成 batch
5. 返回 `(x_batch, y_batch)`

---

# 8. 维度推演：batch 之后 shape 会变成什么

假设单条样本是：

* `x.shape = (3, 224, 224)`
* `y.shape = ()` 或 `(1,)`

设置：

```python
batch_size = 32
```

---

## 情况 1：标签是标量

如果在 `__getitem__` 中你返回的是：

```python
y = torch.tensor(5)
```

那么单条标签 shape 是：

```python
()
```

经过 `DataLoader` 堆叠后：

```python
y_batch.shape == (32,)
```

---

## 情况 2：标签是长度为 1 的张量

如果你返回的是：

```python
y = torch.tensor([5])
```

那么单条标签 shape 是：

```python
(1,)
```

经过堆叠后：

```python
y_batch.shape == (32, 1)
```

---

## 图像 batch 的 shape

无论哪种情况，只要单张图像是：

```python
x.shape == (3, 224, 224)
```

batch 后一定是：

```python
x_batch.shape == (32, 3, 224, 224)
```

这就是视觉任务里最经典的四维格式：

```python
(Batch, Channels, Height, Width)
```

---

# 9. 标准代码模板

```python
import torch
from torch.utils.data import Dataset, DataLoader

# 1. 定义 Dataset
class TrafficSignDataset(Dataset):
    def __init__(self, num_samples):
        # 实际工程中，这里通常保存图片路径、标签路径或元信息
        self.num_samples = num_samples

    def __len__(self):
        return self.num_samples

    def __getitem__(self, index):
        # 实际工程中，这里通常会从硬盘读取第 index 条数据
        # 例如：img = cv2.imread(self.image_paths[index])

        # 这里用随机张量模拟一张 3x224x224 的图片
        x = torch.randn(3, 224, 224)

        # 模拟一个类别标签，返回标量
        y = torch.tensor(5)

        return x, y


# 2. 实例化 Dataset
dataset = TrafficSignDataset(num_samples=1000)

# 3. 实例化 DataLoader
dataloader = DataLoader(
    dataset,
    batch_size=32,
    shuffle=True,
    num_workers=0
)

# 4. 在训练循环中取 batch
for epoch in range(1):
    for batch_idx, (x_batch, y_batch) in enumerate(dataloader):
        print(f"Batch {batch_idx}")
        print(f"x_batch shape: {x_batch.shape}")   # (32, 3, 224, 224)
        print(f"y_batch shape: {y_batch.shape}")   # (32,)
        break
```

---

# 10. 训练循环里为什么写成 `for x_batch, y_batch in dataloader`

因为：

* `Dataset.__getitem__()` 返回的是**单条样本** `(x, y)`
* `DataLoader` 自动把很多条样本整理成一个 batch
* 所以训练时拿到的是 `(x_batch, y_batch)`

于是训练循环自然写成：

```python
for x_batch, y_batch in dataloader:
    y_pred = model(x_batch)
```

---

# 11. `return x, y` 为什么必须是元组

在监督学习里，模型不仅要输入 `x`，还要知道真实答案 `y` 来计算损失。

所以 `__getitem__` 一般返回：

```python
return x, y
```

这样 `DataLoader` 会把：

* 32 个 `x` 堆叠成 `x_batch`
* 32 个 `y` 堆叠成 `y_batch`

最后返回：

```python
(x_batch, y_batch)
```

---

## 如果只返回 `x`

如果你的 `__getitem__` 写成：

```python
return x
```

那 `DataLoader` 最终只会返回一个 batch，而不是两个值。

这时如果你还写：

```python
x_batch, y_batch = next(iter(dataloader))
```

就会报错，因为只有一个东西可解包。

---

# 12. `shuffle=True` 为什么重要

## 训练集

训练时通常应该：

```python
shuffle=True
```

原因是：如果数据按类别排序，模型会连续看到大量同类样本，梯度更新方向会严重偏向当前局部数据，导致训练不稳定，甚至不收敛。

打乱后，每个 batch 的类别分布更随机，梯度更新更平稳。

---

## 验证集 / 测试集

验证和测试通常应该：

```python
shuffle=False
```

因为这时目标不是学习，而是：

* 稳定评估模型
* 保留样本顺序
* 方便定位哪些样本预测错误

---

# 13. `num_workers` 是性能关键参数

如果不写：

```python
num_workers
```

默认通常是：

```python
num_workers=0
```

这意味着：

* 主进程自己负责取数据
* 训练和数据准备更容易串行发生
* GPU 可能经常空等数据

---

## 更高效的做法

在真实工程里，常写：

```python
DataLoader(..., num_workers=4)
```

或者更大。

这样可以让多个 worker **并行准备数据**，在 GPU 计算当前 batch 时，后台已经在准备下一个 batch。

这通常是提高训练吞吐量的关键参数之一。

---

# 14. `collate_fn` 是什么

`DataLoader` 能自动把多个单样本拼成 batch，是因为它内部有一个默认的整理逻辑，可以理解成默认的 `collate_fn`。

例如：

如果单条样本返回：

```python
(x, y)
```

那么 32 条样本会自动变成：

```python
(x_batch, y_batch)
```

---

## 为什么需要自定义 `collate_fn`

如果你的数据不是规整张量，比如：

* 文本序列长度不一样
* 检测任务里每张图目标框数量不同

默认拼接方式就可能失效。
这时就需要自定义 `collate_fn`，告诉 `DataLoader`：

> “一组样本应该如何拼成一个 batch。”

---

# 15. 最常见的形状错误

这是实战中非常重要的坑。

## 坑 1：标签 shape 混淆

单条标签如果是：

```python
torch.tensor(5)
```

batch 后是：

```python
(32,)
```

单条标签如果是：

```python
torch.tensor([5])
```

batch 后是：

```python
(32, 1)
```

这两个 shape 对很多 loss 函数来说含义不同，常常会引发：

* 维度不匹配报错
* 静默逻辑错误
* 训练结果异常

---

## 坑 2：以为 `Dataset` 返回的是 batch

不是。`Dataset` 只负责**单样本**。
batch 是 `DataLoader` 负责组织出来的。

---

## 坑 3：以为 `DataLoader` 只是在外面套了个 `for`

也不是。
它本质上还负责：

* batch 组织
* shuffle
* worker 并行
* collate 逻辑

它是一个“数据调度器”，不是简单语法糖。

---

# 16. 从设计者视角的最终理解

## `Dataset`

定义的是：

> **样本的语义和单样本读取逻辑**

它关心的是：

* 数据在哪
* 一条数据怎么读
* 一条数据怎么变成张量

---

## `DataLoader`

定义的是：

> **样本如何高效、稳定地流入训练循环**

它关心的是：

* 顺序
* batch
* 并行
* 拼接
* 吞吐效率

---

# 17. 最终总结

**在 PyTorch 中，`Dataset` 负责定义“一个样本是什么、怎么读出来”，`DataLoader` 负责定义“许多样本怎样高效地组成 batch 并喂给模型”。**
