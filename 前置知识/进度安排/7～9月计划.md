### Day 2 ｜ Node（节点）与 Topic（话题）

**理论目标**
- [ ] 🧠 L2 理解 ⭐：ROS 2 的核心概念

| 概念 | 类比 | 含义 |
|------|------|------|
| Node（节点） | 一个工人 | 独立运行的进程，负责一个具体功能 |
| Topic（话题） | 公告板 | 节点之间传递消息的"频道" |
| Publisher（发布者） | 在公告板上贴通知 | 往 Topic 上发送数据 |
| Subscriber（订阅者） | 看公告板的人 | 从 Topic 上接收数据 |
| Message（消息） | 通知的内容 | Topic 上传输的数据结构 |

- [ ] 🧠 L2 理解：发布-订阅模式的好处 — 发布者和订阅者互不知道对方存在（解耦）
- [ ] 🧠 L2 理解：一个 Topic 可以有多个 Publisher 和多个 Subscriber

**实践目标**
- [ ] 💻 L2 能照写：用命令行查看系统中的话题和节点


### Day 3 ｜ 第一个 Python 节点：Publisher

**理论目标**
- [ ] 🧠 L2 理解：ROS 2 节点的生命周期 — 创建 → 运行（spin）→ 销毁
- [ ] 🧠 L1 知道：`rclpy` 是 ROS 2 的 Python 客户端库

**实践目标**
- [ ] 💻 L3 能默写 ⭐：一个最简单的 Publisher 节点

```python
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class MinimalPublisher(Node):
    def __init__(self):
        super().__init__('minimal_publisher')      # 节点名
        self.publisher_ = self.create_publisher(
            String, 'chatter', 10)                 # 消息类型, 话题名, 队列长度
        self.timer = self.create_timer(
            0.5, self.timer_callback)               # 每 0.5 秒触发一次
        self.count = 0

    def timer_callback(self):
        msg = String()
        msg.data = f'Hello ROS2: {self.count}'
        self.publisher_.publish(msg)
        self.get_logger().info(f'Publishing: {msg.data}')
        self.count += 1

def main():
    rclpy.init()
    node = MinimalPublisher()
    rclpy.spin(node)            # 保持节点运行
    node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()
```

- [ ] 💻 L2 理解每一行的作用：
  - `create_publisher(类型, 话题名, 队列)` — 创建发布者
  - `create_timer(间隔, 回调)` — 定时触发
  - `rclpy.spin(node)` — 让节点持续运行，等待事件

**自测**：如果想把发布频率从 2Hz 改成 10Hz，改哪个数字？（答案：`0.5` 改成 `0.1`）

### Day 2–3 ｜ YOLO + ROS 2 感知节点 ⭐

**理论目标**
- [ ] 🧠 L2 理解：感知节点的数据流

```
/camera/image_raw → [你的感知节点] → /perception/detections
     (输入图像)       (YOLO 推理)        (检测结果)
                                      → /perception/viz_image
                                         (可视化图像，调试用)
```

- [ ] 🧠 L2 理解：节点需要在回调函数中完成：收图 → 预处理 → 推理 → 后处理 → 发布结果

**实践目标**
- [ ] 💻 L4 能魔改 ⭐：编写完整的 YOLO 感知节点

```python
import rclpy
from rclpy.node import Node
from sensor_msgs.msg import Image
from vision_msgs.msg import Detection2DArray, Detection2D, ObjectHypothesisWithPose
from cv_bridge import CvBridge
from ultralytics import YOLO

class YoloDetectionNode(Node):
    def __init__(self):
        super().__init__('yolo_detection_node')
        
        # 订阅图像
        self.subscription = self.create_subscription(
            Image, '/camera/image_raw', self.image_callback, 10)
        
        # 发布检测结果
        self.det_publisher = self.create_publisher(
            Detection2DArray, '/perception/detections', 10)
        
        # 发布可视化图像（调试用）
        self.viz_publisher = self.create_publisher(
            Image, '/perception/viz_image', 10)
        
        self.bridge = CvBridge()
        self.model = YOLO('best.pt')
        self.get_logger().info('YOLO Detection Node started.')

    def image_callback(self, msg):
        # 1. ROS Image → OpenCV
        cv_image = self.bridge.imgmsg_to_cv2(msg, 'bgr8')
        
        # 2. YOLO 推理
        results = self.model(cv_image, verbose=False)
        
        # 3. 构建检测消息
        det_array = Detection2DArray()
        det_array.header = msg.header
        
        for r in results[0].boxes:
            det = Detection2D()
            # 边界框
            x1, y1, x2, y2 = r.xyxy[0].tolist()
            det.bbox.center.position.x = (x1 + x2) / 2
            det.bbox.center.position.y = (y1 + y2) / 2
            det.bbox.size_x = x2 - x1
            det.bbox.size_y = y2 - y1
            # 类别和置信度
            hyp = ObjectHypothesisWithPose()
            hyp.hypothesis.class_id = str(int(r.cls[0]))
            hyp.hypothesis.score = float(r.conf[0])
            det.results.append(hyp)
            det_array.detections.append(det)
        
        # 4. 发布检测结果
        self.det_publisher.publish(det_array)
        
        # 5. 发布可视化图像
        viz = results[0].plot()
        self.viz_publisher.publish(self.bridge.cv2_to_imgmsg(viz, 'bgr8'))
        
        self.get_logger().info(f'Detected {len(det_array.detections)} objects')
```

- [ ] 💻 L2 能照写：能逐行解释这段代码中每一步在做什么

### Day 2 ｜ QoS（服务质量配置）

**理论目标**
- [ ] 🧠 L2 理解：QoS = 控制消息传输的可靠性和行为策略
- [ ] 🧠 L2 理解：两种关键 QoS 策略

| 策略 | 含义 | 何时用 |
|------|------|--------|
| Reliable | 保证消息到达（丢了会重发） | 控制指令等不能丢的数据 |
| Best Effort | 尽力而为（丢了不重发） | 图像等高频数据，允许偶尔丢帧 |

- [ ] 🧠 L2 理解：图像传输通常用 Best Effort — 丢一帧没关系，但要保持实时性
- [ ] 🧠 L1 知道：Publisher 和 Subscriber 的 QoS 必须兼容，否则收不到消息（常见 bug）

**实践目标**
- [ ] 💻 L2 能照写：给图像订阅设置 Best Effort QoS

```python
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy

qos_profile = QoSProfile(
    reliability=ReliabilityPolicy.BEST_EFFORT,
    history=HistoryPolicy.KEEP_LAST,
    depth=1  # 只保留最新一帧
)

self.subscription = self.create_subscription(
    Image, '/camera/image_raw', self.image_callback, qos_profile)
```

- [ ] 💻 L1 能读懂：遇到"订阅了但收不到消息"时，首先检查 QoS 是否匹配

### Day 3 ｜ Service 与 Action 概念

**理论目标**
- [ ] 🧠 L2 理解：Topic vs Service vs Action 的区别

| 通信方式 | 模式 | 适用场景 | 类比 |
|---------|------|---------|------|
| Topic | 单向持续流 | 传感器数据、感知结果 | 广播电台 |
| Service | 请求-响应 | 触发一次性操作（如保存地图） | 打电话 |
| Action | 长时间任务 + 反馈 | 导航到某点（耗时+进度） | 外卖下单 |

- [ ] 🧠 L1 知道：感知模块主要用 Topic，但可能用 Service 做"切换模型"等操作
- [ ] 🧠 ⚠️ 不需要深入实践：了解概念即可，日常感知开发以 Topic 为主

## Week 30 · TensorRT 加速

### Day 1 ｜ TensorRT 概念

**理论目标**
- [ ] 🧠 L2 理解 ⭐：TensorRT 是 NVIDIA 专属的推理优化引擎
- [ ] 🧠 L2 理解：TensorRT 为什么比 ONNX Runtime 还快 — 因为它针对 NVIDIA GPU 做了深度优化

| 优化手段 | 含义 |
|---------|------|
| 层融合（Layer Fusion） | 把多个小算子合并成一个大算子，减少内存读写 |
| 精度降低（FP16/INT8） | 用低精度计算换取速度，牺牲少量精度 |
| 核自动调优（Kernel Auto-tuning） | 对每个算子试跑多种 CUDA 实现，选最快的 |
| 动态 Tensor 内存 | 优化 GPU 内存分配，减少碎片 |

- [ ] 🧠 L2 理解：精度等级

| 精度 | 位数 | 速度 | 精度损失 |
|------|------|------|---------|
| FP32 | 32位 | 基准 | 无 |
| FP16 | 16位 | ~2× | 通常可忽略 |
| INT8 | 8位 | ~4× | 需要标定，可能有损失 |

- [ ] 🧠 L1 知道：车端常用 FP16，INT8 需要额外标定步骤

**实践目标**

## Week 33 · Transformer 在视觉中的应用

### Day 1 ｜ Attention 机制直觉

**理论目标**
- [ ] 🧠 L2 理解 ⭐：Attention = "让模型决定该关注图像/序列的哪个部分"
- [ ] 🧠 L2 理解：Self-Attention 的直觉 — 每个位置和所有位置计算"关联度"，关联度高的给更多权重
- [ ] 🧠 L2 理解：Q/K/V 的类比
  - Query（查询）= "我在找什么"
  - Key（键）= "我有什么"
  - Value（值）= "我能提供什么"
  - Attention(Q,K,V) = softmax(Q×Kᵀ/√d) × V
- [ ] 🧠 ⚠️ 不需要 L3：不需要手推多头注意力的完整公式，理解直觉即可

### Day 2 ｜ ViT 与视觉 Transformer

**理论目标**
- [ ] 🧠 L2 理解：ViT（Vision Transformer）的核心思想 — 把图像切成 patch，每个 patch 当作一个 "token"

```
图像 → 切成 16×16 的 patch → 每个 patch 展平为向量 → 送入 Transformer → 分类
```

- [ ] 🧠 L2 理解：ViT vs CNN 的核心区别
  - CNN：通过卷积逐层扩大感受野，擅长局部特征
  - ViT：通过 Attention 一次看到所有 patch，擅长全局关系
- [ ] 🧠 L2 理解：ViT 的劣势 — 需要更多数据才能训好（因为没有 CNN 的归纳偏置）

### Day 3 ｜ 从 2D 到 3D：为什么需要 BEV

**理论目标**
- [ ] 🧠 L2 理解 ⭐：传统 2D 感知的局限

| 问题 | 说明 |
|------|------|
| 没有深度 | 2D 检测框不知道目标离你多远 |
| 多相机不统一 | 每个相机各自检测，结果怎么融合？ |
| 遮挡难处理 | 前视图中重叠的物体无法区分先后 |
| 规控需要 3D | 规划路径需要知道目标的 3D 位置和速度 |

- [ ] 🧠 L2 理解 ⭐：BEV 的核心思想 — 把所有相机的 2D 图像"提升"到一个统一的 3D 俯视空间

```
前视相机 ─┐
后视相机 ─┤
左视相机 ─┼→ [2D→3D 视角变换] → BEV 特征图 → 3D 检测/分割
右视相机 ─┤                       (俯视网格)
...      ─┘
```

- [ ] 🧠 L2 理解：BEV 的优势
  - 多相机结果在同一个空间中融合，不再需要手工匹配
  - 直接输出 3D 信息（位置、朝向、速度）
  - 更容易接入下游规控模块

### Day 4 ｜ BEV 感知核心方法

**理论目标**
- [ ] 🧠 L2 理解：2D → 3D 的两种主流思路

| 方法 | 代表模型 | 核心思想 |
|------|---------|---------|
| 显式深度估计 | BEVDet / LSS | 先估计每个像素的深度，再用深度把 2D 特征"铺"到 3D 空间 |
| 隐式 Attention | BEVFormer | 用 Transformer 的 Cross-Attention 直接查询 2D 特征，学习 3D 空间映射 |

- [ ] 🧠 L2 理解：LSS（Lift-Splat-Shoot）的直觉
  1. Lift：给每个像素预测"可能在不同深度的概率"
  2. Splat：把特征按深度概率散布到 3D 体素空间
  3. Shoot：从俯视角度压缩得到 BEV 特征
- [ ] 🧠 L2 理解：BEVFormer 的直觉
  1. 在 BEV 平面上定义一组查询点
  2. 每个查询点用 Attention 到多个相机的 2D 特征中"找"对应信息
  3. 不需要显式估计深度
- [ ] 🧠 L1 知道：时序融合 — BEVFormer 还可以用历史帧的 BEV 特征做 Temporal Attention
